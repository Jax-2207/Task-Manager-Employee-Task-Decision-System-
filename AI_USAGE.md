# AI Usage & Guidance

This document describes how AI tools were used during development, how the generated code was reviewed, and what constraints were placed on AI behavior — as required by the assessment.

---

## AI Tools Used

- **AI Coding Assistant** — Used for code generation, refactoring, and debugging across both backend and frontend.

## How AI Was Constrained

All AI-generated code was governed by the **`backend/ai_guidance/agents.md`** file, which enforces:

| Rule Category | What It Prevents |
|---|---|
| **Architecture boundaries** | Business logic leaking into routes, DB access outside repositories |
| **Domain safety** | Invalid state transitions (e.g., ACCEPTED → REJECTED) |
| **Change management** | Modifying unrelated files, breaking existing signatures |
| **Simplicity** | Over-engineered patterns, premature abstraction, magic decorators |
| **Testing** | Skipping tests, missing negative test cases |

The AI was explicitly instructed: *"This is NOT a feature-building task. This is a system integrity task."*

## How Generated Code Was Reviewed

Every AI-generated change was verified against these criteria before acceptance:

1. **Does it violate architectural boundaries?** — Routes must not contain business logic; repositories must not enforce domain rules.
2. **Does it break existing tests?** — All 18 tests must pass after every change (`pytest tests/ -v`).
3. **Does it introduce unnecessary complexity?** — If a simpler solution exists, the complex one is rejected.
4. **Does it handle errors explicitly?** — Silent failures are not acceptable; domain exceptions must propagate.
5. **Does it preserve the state machine?** — The Decision/Task status transitions (PROPOSED → ACCEPTED/REJECTED) must remain centrally enforced in the service layer.

## What Was AI-Generated vs. Human-Decided

| Aspect | Who Decided |
|---|---|
| Architecture (layered boundaries) | Human — defined in `agents.md` before any code was written |
| State transition rules | Human — specified as domain invariants |
| Authentication strategy (Google + hardcoded admin) | Human — chosen to avoid service account complexity |
| Code implementation | AI — generated under `agents.md` constraints |
| Test cases (including negative cases) | AI — generated, then reviewed for coverage |
| WhatsApp API endpoint design | Human — decided public endpoint was acceptable for local-network use |
| README and documentation | Collaborative — structure decided by human, content generated by AI |

## Risks of AI Usage

1. **Over-generation** — AI tends to produce more code than necessary. This was mitigated by the simplicity rules in `agents.md`.
2. **Incorrect assumptions** — AI may assume requirements that don't exist. Every generated feature was validated against the original assessment brief.
3. **Security blind spots** — AI-generated auth code was manually reviewed for token handling, input validation, and error exposure.
